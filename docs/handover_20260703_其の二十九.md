## 実施内容の詳細

### 1. 方針決定の経緯

其の二十八で発見した①②（bui自己申告のプロンプト/ルールエンジン非同期）の解消方針として、セッション冒頭で3案を検討した：

- 方針A：kuzari_rules.yml等に虫・鳥・花等を独立カテゴリとして追加
- 方針B：ShikimokuChecker内部に正規化ロジックを直接実装
- 方針C：辞書層（B層）で正規化し、ShikimokuChecker呼び出し前に適用

`docs/minase_soron.md`の「仕様1　体用フラグの必須化」を確認したところ、アーキテクチャ図で「部立の辞書照合（体用フラグ付き）」が明確にB層（Ruby+辞書）の責務とされており、A層（ShikimokuChecker）は「辞書照合済みの結果を受け取る純粋なガードレール」という役割分担が既に設計されていた。また`kuzari_violations`は既に`bui_dict:`引数を体用判定のために受け取る仕組みが存在した。この先例に従い**方針C**を採用した。

方針Aは「虎（動物）を独立カテゴリとして扱う」という意味的な不正確さがあり、方針Bは大原則「ShikimokuCheckerは純粋関数」との緊張関係があったため、いずれも採らなかった。

### 2. 実装

- `app/data/bui_dictionary.yml`: 「虫」「鳥」「獣」を`primary_bui: 動物`として新規追加（②の紅葉・桜・柳等は既に登録済みだったため追加不要）
- `app/services/bui_dictionary.rb`: `normalize_bui(tag, valid_categories)`を追加。正規カテゴリならそのまま、辞書の`primary_bui`で解決できればそれに置換、辞書にも無ければそのまま返す（③は無変換のまま、ShikimokuChecker側で従来通り無視される仕様を維持）
- `script/dryrun_hyakuin.rb`: `parse_candidate`直後（duplicate_verse判定より前）でbui正規化を適用。正規化前の自己申告は`raw_bui`として保持し、`generation_attempts.log`に両方記録（③の推移を将来も監視できるようにするため）。`HAKKU`（発句の固定初期値、`bui: ["花","植物"]`）も同様に正規化した上で`history`に格納
- ShikimokuChecker・kuzari_rules.yml・kukazo_rules.ymlは**無変更**

### 3. 検証

- 単体確認：①②に該当する12語すべてが正しく正規化されることを確認
- 実効性確認：正規化前は「虫」と「鳥」が別々の未知タグとして句去チェックをすり抜けていたが、正規化後はどちらも「動物」として扱われ、5句去違反が正しく検出されることを確認（`kuzari_violations`への直接呼び出しで再現）
- 回帰テスト：`script/verify_shikimoku.rb`試験13として19件追加、**88 pass / 0 fail**
- 実環境検証：実際にOllama経由で100句のdryrunを実行し完走、generation_failed 0件。今回実行分226候補を対象に、正規化前(raw_bui)の形式ng率20.0%→正規化後(bui)6.8%への低下を実測。残る6.8%は③（真に未知）に相当し、想定通り監視継続対象として残る

### 4. 中優先項目1：残存ケース再検討基準の言語化

其の二十七の観測値（900句中1件=0.111%）についてWilson信頼区間（95%）を計算：**0.020%〜0.627%**。これを踏まえた3段階基準を策定した：

1. **監視継続ライン（現状）**: 発生率が0.6%以下 → 其の二十七時点の想定からの逸脱とは言えず、対処不要
2. **注意ライン**: 累積500句以上のサンプルで発生率が0.6%を超えた場合 → 統計的な悪化とみなし、`analyze_generation_failures.rb`で原因分析に着手
3. **着手ライン（(b)本格的seed pool新設の検討開始）**: 上記の悪化が2回以上の独立したdryrun実行にわたって再現された場合、または同一原因（duplicate_verse固着で救済発動後も全滅、等）による事例が3件以上累積した場合 → 単発のノイズでなく構造的な問題とみなし、(b)の設計検討を開始する
