# 引き継ぎ文書：其の五十八（2026-07-24）

## 位置づけ

其の五十七で判明した「七句去物」ルール（去嫌一覧表の「七」列の正体）が、
宗祇関連の実作（水無瀬三吟・湯山三吟・遺誡百韻）で実際にどの程度遵守されて
いたかを実証検証し（Phase 0）、あわせて未登録語彙の辞書拡充を行った
（Phase 1）セッション。Phase 0の結果は「実装価値あり」の強いシグナルだったが、
Phase 2の実装設計中に無関係の重大な既存バグ（`ichiza_violations`が本番で
実質機能していない）を発見したため、Phase 2はドラフト提示のみに留め、
適用は見送った（人間の判断）。

## ゲートチェック

セッション開始時・Phase 1適用後ともに `88 pass / 0 fail` を維持。

## Phase 0：七句去物ルールの実証検証

対象10語（衣・竹・田・船・夢・涙・月・松・枕・煙）について、3作品の全出現
句番号を洗い出し、同一語の再出現間隔を検算した。

**間隔の数え方：** 本コードベースの既存の句去チェック（`kuzari_violations`の
`between = n - j`、`between >= interval - 1`で合法）を代数的に展開すると、
「候補句の句番号 − 直前出現句の句番号 ≧ interval」で合法、という単純な
句番号差の比較に一致する。以下の表もこの規約（句番号の生の差）で統一した。

**月の42句「月日の末や夢にめぐらむ」は、勢田論文が明記する適用除外
（「年月」「月日」等の月次用法は七句去物の対象外、光物としての「月」のみが
対象）に該当する。** この除外規定を適用すると42句は集計から除外され、
月の出現句番号は [5,18,27,48,58,66,90]、最小間隔13句となり違反は消える。
**除外規定を適用せず単純に「月」の文字列一致だけで数えた場合のみ、
27句→42句は間隔15句でセーフだが、42句→48句が間隔6句で唯一の「違反」候補
となる。** つまり、この一件は「規定を破った例」ではなく、**むしろ
「月」の字面を使いたいが7句以内で直前に光物の月を使ったばかりなので、
月次表現（月日）に言い換えて回避した実例**と解釈するのが妥当であり、
七句去物ルールが実際に意識的に運用されていたことを示す傍証と言える。

枕の18句→89句は間隔ちょうど7句（規定は「7句以上」なので境界値でセーフ）。
これも偶然というより、規定の下限ぎりぎりを意識した運用の可能性がある。

### 作品間の傾向差

独吟（宗祇一人で詠む）の遺誡百韻と、連衆吟（3名）の水無瀬三吟・湯山三吟との
間で、七句去物の遵守度に差は見られなかった。独吟だからといって規定が緩む
様子はない。連衆吟でも各詠み手が前句までの履歴を把握した上で七句去物を
守っていたことになり、当時の連歌師にとってこの規定が暗黙の共通認識
だったことをうかがわせる。

### 判断：Phase 2（実装）に進む価値あり

3作品・10語・全ペア（湯山三吟9ペア、遺誡百韻7ペア、水無瀬三吟14ペア、
計30ペア）を検算した結果、**「7句未満で真に違反」と言える事例は0件**
だった（水無瀬三吟の1件は除外規定の適用対象であり、むしろ規定遵守の
証拠）。依頼書の判断基準に照らし、Phase 2（ShikimokuChecker実装）に
進む価値があると判断した。

## Phase 2：ShikimokuCheckerへの七句去物ルール実装 → ドラフト提示のみ、適用見送り

### 見送りの理由：ichiza_violationsの本番無効化バグを発見

Phase 2の設計中、既存の`ShikimokuChecker#ichiza_violations`（一座一句物
チェック）を七句去物実装のテンプレートにしようとして、`app/controllers/
rengas_controller.rb`と`script/observe_production_hyakuin.rb`の両方を
確認したところ、**`history`・`candidate`のいずれにも`:word`キーが
一度も設定されていない**ことが判明した。

```ruby
# rengas_controller.rb#create（本番経路）
candidate = {
  bui:        bui_dict.detect_all(tsugeku, nm),
  season:     season_from_text(tsugeku),
  verse_type: next_verse_type
}
# :word キーが存在しない
violations += checker.ichiza_violations(history, candidate)
```

```ruby
# app/controllers/rengas_controller.rb#build_verse_history
{ bui: bui_dict.detect_all(text, nm), season: season_from_text(text), verse_type: vtype }
# こちらにも :word キーが存在しない
```

`ichiza_violations`は`verse[:word].to_s`への部分文字列マッチで一座一句物語の
重複を検出する設計だが、`:word`が常に`nil`（`.to_s`で`""`）になるため、
**`text.include?(iw)`は空文字列同士の比較になり、実際には一度も
`ichiza_duplicate`を検出できない状態のまま本番稼働している。**
`script/observe_production_hyakuin.rb`側も同一の構造で同じ問題を抱える。

これは今回の七句去物タスクとは無関係の、既存の一座一句物チェック機能
（おそらく其の二十四前後で実装）に関する独立した発見であり、本セッションの
依頼書スコープ外のため**修正は行っていない**。

### 判断：Phase 2はドラフトのみ提示、適用は見送り（人間の判断）

`ichiza_violations`と同じ`verse[:word]`部分一致方式で七句去物を実装しても、
配線ギャップが解消されない限り本番では同様に無効化された状態になる。
テストコード（`verify_shikimoku.rb`）上は正しく動作して見えるが、実運用に
一切影響しない「見せかけの実装」になるリスクがあるため、人間の判断で
**今回はドラフト提示のみとし、実ファイルへの適用は行わなかった。**

**2. `app/services/shikimoku_checker.rb`（ドラフト差分）**

`ichiza_violations`と対になる`nanaku_gomono_violations`を追加。ichiza
（一座一句物＝二度目は常に違反）と異なり、直近の再出現位置からの間隔が
7句未満の場合のみ違反とする設計。

```ruby
NANAKU_GOMONO_INTERVAL = 7

def nanaku_gomono_violations(history, candidate, nanaku_words = @nanaku_words)
  cand_text = candidate.is_a?(Hash) ? candidate[:word].to_s : candidate.to_s
  last_seen = {}
  history.each_with_index do |verse, i|
    text = verse.is_a?(Hash) ? verse[:word].to_s : verse.to_s
    Array(nanaku_words).each { |nw| last_seen[nw] = i + 1 if text.include?(nw) }
  end
  violations = []
  Array(nanaku_words).each do |nw|
    next unless cand_text.include?(nw)
    pos = last_seen[nw]
    next unless pos
    interval = history.size + 1 - pos
    next if interval >= NANAKU_GOMONO_INTERVAL
    violations << { type: :nanaku_gomono, word: nw, last_pos: pos,
                     pos: history.size + 1, interval: interval }
  end
  violations
end

def nanaku_gomono_ok?(history, candidate, nanaku_words = @nanaku_words)
  nanaku_gomono_violations(history, candidate, nanaku_words).empty?
end
```

ロジックは一時的なモンキーパッチでサニティチェック済み（リポジトリ非変更）：
月が1句目に既出・候補7句目（差6）→違反検出／候補8句目（差7）→違反なし
（境界値セーフ）を確認。既存`kuzari_violations`の`between = n - j`,
`between >= interval - 1`規約を代数展開した結果と一致する設計であることも
確認済み。

**3. `script/verify_shikimoku.rb`（試験14ドラフト）**

`ichiza_violations`の試験10と同じ形式で、境界値ケース（差6→違反／差7→
セーフ）と水無瀬三吟全100句chain走査（クラッシュしないことのみ確認、
`word:`フィールドが短縮タグのためPhase0のような実データ検証はできない

全ドラフトの完全なテキストは本セッションの会話ログに記録済み。次回実装時は
そこから復元可能。
