## 多段階付句生成の設計構想（将来フェーズ）

### 背景
mora_error固着対策の検討中に、連歌の付け方の本来の構造に
近い多段階生成の構想が浮上した。独吟（AI単独）から
人間参加（renga-kasen-mail等）への移行時に必要となる
機能の道筋として記録する。


### 提案された生成フロー
前句
→ ①A層/B層：前句から次句に使える季語・花鳥風月の候補を抽出
→ ②C層（LLM）：候補語を使って自然な和歌（31音相当）を生成
→ ③Ruby/KuValidator：和歌を上句（17音）/下句（14音）に分割
前句が長句 → 下句（14音）を付句として採用
前句が短句 → 上句（17音）を付句として採用
→ ④式目チェック → 記録


LLMの責務は「季語候補を使って自然な和歌を詠む」のみ。
モーラ計算・切り出し・式目判定はRubyが担当。
役割分担がCOBOLの共通サブルーチン化と同じ構造で明快。

### ①の実装道筋（A層/B層拡張）

| Phase | 内容 | 依存 |
|---|---|---|
| Phase 1 | 前句のseason特定 → 次句で使える季の制約を返す（ShikimokuCheckerの逆引き） | 既存ロジック流用 |
| Phase 2 | BuiDictionaryから該当季・部立の語を候補リストとして返す | 既存YAMLを読み込む形で実装可能 |
| Phase 3 | 人間の付句が式目違反の場合に「この季語なら成立する」と返す | renga-kasen-mail設計と連動 |

Phase 1・2は現在の実装の延長で対応可能。
Phase 3はrenga-kasen-mailキックオフ時に合わせて設計する。

### 独吟から対人へ移行する際に必要な機能

現在のwaka-collectorは独吟（AI単独の百韻生成）を前提としている。
人間が参加者として加わる場合（renga-kasen-mail等）には
以下の機能が新たに必要になる。

**A. 人間の付句を受け取り評価する機能**
- KuValidator：字数チェック（既存）
- ShikimokuChecker：式目チェック（既存）
- 上記2つは既に実装済みだが、現状は生成後の表示にのみ使用
  → 人間の句を「ゲートとして通す/差し戻す」判定ロジックへの転用が必要

**B. 差し戻し時の案内機能**
- 式目違反の理由を人間が理解できる言葉で説明する
  （現状のstyle_check_result["violations"]をそのまま表示するだけでは不親切）
- 「こういう季語を使えば成立する」という代替候補の提示
  → Phase 1・2の実装が前提

**C. 進行管理機能**
- 誰の番か（座の管理）
- タイムアウト・催促（renga-kasen-mailではメール送信で代替）
- 完成した百韻の保存・表示

**D. 式目の人間向け可視化**
- 現在どの季が何句続いているか
- 次の句で使える季語・部立の制約サマリー
- 花・月の出現状況（あと何句使えるか）

**E. AIとの協調生成**
- 人間が詠めない場合にAIが補完する「助け舟」機能
- この場合のAI生成は多段階フロー（上記②③）が望ましい

## 守るべき大原則（再掲）

1. **枯れてから足す** — 実害が観測されてから対処する。観測前の予防的追加は禁止。
2. **骨法は Ruby、即興はメンタムさん** — 式目判定・ループ制御は Ruby、付句生成は LLM。
3. **ShikimokuChecker は純粋関数** — 外部状態・DB 参照を持ち込まない。
4. **正解データは水無瀬三吟** — テストの期待値は水無瀬三吟に準拠する。
5. **D-19-1 変更禁止** — モーラ許容 ±1 設計は意図的設計。厳格化しない。
6. **D-19-5 変更禁止** — 句数上限の Ruby 側強制切り換え設計。変更しない。
7. **D-22-2 変更禁止** — ひらがな神祇語の観測記録。追記のみ許可、削除・変更禁止。
8. **機械的救済の中身は式目・連歌の作法に忠実であること** — 季語ランダム変更のような式目無視の対症療法は別の式目違反を誘発しうる（其の二十七で実証済み）。
9. **辞書照合ロジックは部分文字列一致でなく形態素解析ベースで行うこと** — 其の二十八で得た教訓。

### 今回の教訓

**フィードバックの「親切さ」が必ずしも生成品質を上げるとは限らない。** Step B-2（yomi裁ち切り）は目標モーラ数により近い情報をLLMに与える意図だったが、不完全な読みの提示がかえって8Bモデルの認知負荷を上げ、固着型の生成失敗を増加させた。フィードバック設計の改善は、追加した情報量ではなく実測（dryrun）で効果を検証すること。
