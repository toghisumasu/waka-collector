# Phase 8-3 設計判断メモ ― 折末・折立と式目バージョンの着せ替え

**位置づけ:** `ShikimokuChecker` 拡張に着手する前の設計判断記録
**目的:** 折末・折立ルールをどこまで実装するかの方針確定と、将来の式目バージョン切り替えの足がかり
**作成:** 2026-06-26
**前提テスト状況:** HEAD a2e7598 / 37 pass / 0 fail

---

## 1. 結論（先に書く）

1. **水無瀬三吟（宗祇・1488）を唯一のデフォルト式目とする。**
2. **折末・折立に新しい強制ルールは足さない。** 折跨ぎでの句去緩和（折去）も折跨ぎ制限（折跨ぎの禁）も、現時点では実装しない。
3. ただし将来の拡張に備え、**式目バージョンを差し替えられる「器」だけ用意する。** 中身は空のまま。
4. これは「枯れてから足す」の原則そのものである。2つ目の式目バージョンが実際に要求されるまで、ロジックは書かない。

---

## 2. なぜ折末・折立ルールを足さないのか

### 2-1. 一次資料の確認結果（裏取り済み）

| 項目 | 確認結果 |
|:---|:---|
| 応安新式（1372・二条良基） | 式目の集大成。**折末・折立の明文規定は本文に存在しない** |
| 新式今案 | 一条兼良の著作（※ Gemini は宗祇の著作と誤記） |
| 折立（おりたて／おりたち） | 用語としては実在。「各折の最初の句」。初折第一句は発句として別扱い |
| 紹巴『連歌至宝抄』『式目秘抄』(1587) | この時期に式目が精緻化された |
| 江戸期 | 幕府連歌として固定化、折跨ぎ制限が硬直化 |
| 「折端」総称・『連歌新式問答』 | **未確認（要一次資料）。現時点では採用しない** |

### 2-2. 水無瀬三吟の実データが示すこと

折境界7箇所の前後句を実際に確認した結果（`minase_sangin_hyakuin.md` より）：

| 境界 | 折末句 | 折立句 | 観察 |
|:---|:---|:---|:---|
| 初折表/裏 | 8 雑・居所 | 9 秋・山類 | 転換 |
| 初折裏/二折表 | 22 雑・述懐 | 23 雑・述懐 | **述懐が継続** |
| 二折表/裏 | 36 恋 | 37 恋 | **恋が継続** |
| 二折裏/三折表 | 50 冬・降物 | 51 冬・動物 | **冬が継続** |
| 三折表/裏 | 64 雑・植物 | 65 雑・時分恋体 | 雑継続 |
| 三折裏/名残表 | 78 春・聳物 | 79 春・動物 | **春が継続** |
| 名残表/裏 | 92 雑・人倫 | 93 雑・述懐 | 雑継続 |

**観察結論：** 折をまたいで恋・述懐・冬・春が平然と継続している。
後世（紹巴〜江戸期）に強化された「折跨ぎでは部立を転換せよ」という制限は、
**宗祇の時点（1488）では適用されていない。**

### 2-3. 年代による裏付け

- 水無瀬三吟：**1488年（宗祇）**
- 紹巴の式目精緻化：**1587年**
- → 約 **100年の開き**。

折跨ぎの厳格化は紹巴以降の現象であり、正解データである水無瀬三吟には
時代的にも実データ的にも適用されない。これは推測ではなく、
一次資料と実データの双方が一致して示す事実である。

## 3. 式目バージョン着せ替えの設計（器だけ作る）

### 3-1. 基本思想

COBOL 的に言えば「実行時に式目テーブルを差し替えるパラメタライズ」。
コアの判定ロジック（`ShikimokuChecker`）は1つだけ。
ルールを規定する **YAML データセットのみを切り替える。**

### 3-2. 想定ディレクトリ構造（将来像・今は作らない）

```
app/data/shikimoku/
  minase/              # 水無瀬版（宗祇・1488）= デフォルト
    kuzari_rules.yml
    kukazo_rules.yml
    boundary.yml       # 折跨ぎ制限なし（= 空ルール）
  jouha/               # 紹巴版（1587）= 将来
    boundary.yml       # 折跨ぎ厳格化
  edo/                 # 江戸幕府連歌版 = 将来
    boundary.yml       # 折跨ぎの禁・完全固定
```

### 3-3. 呼び出しインタフェースのイメージ

```ruby
# デフォルト（水無瀬版）
checker = ShikimokuChecker.new(ruleset: :minase)

# 将来、紹巴版が要求されたとき
checker = ShikimokuChecker.new(ruleset: :jouha)
```

ルールセット名を引数で受けるだけ。
コアの判定ロジックは一切変更しない。

### 3-4. boundary フラグの扱い（スタブのみ）

将来 `scan_chain` 系メソッドに `oritachi:` / `orisue:` の引数を
受け付ける余地を残してもよいが、**デフォルトは何もしない（no-op）**。

```ruby
# イメージ（実装はしない・スタブのみ）
def boundary_violations(chain, ruleset: @ruleset)
  return [] if ruleset == :minase   # 水無瀬版は折跨ぎ制限なし
  # jouha / edo のロジックは将来ここに足す（今は書かない）
end
```

## 4. 着手条件（いつ中身を書くか）

以下のいずれかが起きるまで、boundary ロジックは空のままにする。

1. ユーザーが「紹巴版／江戸版の式目で検証したい」と明示的に要求した
2. 水無瀬以外の実作（紹巴の百韻など）を正解データとして入手した
3. 折跨ぎで実際に誤判定（偽陽性・偽陰性）が観測された

→ どれも起きていない現時点では、**器（ディレクトリ構造とフラグの口）だけ**用意し、
ロジックはゼロのままで「枯らす」。

---

## 5. このメモの要旨（一行で）

> 水無瀬版で統一し、折末・折立の中身は書かない。
> ただし将来の式目着せ替えに備え、ルールセットを差し替える「口」だけ残す。
