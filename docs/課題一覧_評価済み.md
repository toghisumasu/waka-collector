### #1　`ichiza_violations`の配線バグ

**該当箇所：** `app/controllers/rengas_controller.rb#create`（本番）・
`#build_verse_history`、`script/observe_production_hyakuin.rb`（観測）。

**影響範囲の推定（想定より広範囲であることが判明）：** `history`/`candidate`
ハッシュに`:word`キーが無いのは事実だが、調査の結果、**同じ`:word`欠落は
`ichiza_violations`だけでなく、本番で毎句必ず呼ばれる`all_violations`
（→`kuzari_violations`）内の**体用（taiyo）区別ロジック**にも波及している**
ことを実測で確認した。さらに`:plant_type`キーも同様に欠落しており、
**植物の異種間句去（D-16、花↔草等は3句去でよいとする規定）も本番では
一度も適用されていない**ことを実測で確認した（下記「その他所見」参照）。
影響は「一座一句物が検出されない」という機能欠落に留まらず、**「体用が
異なるので許容されるべき候補」「植物の異種なので短い間隔でよいはずの候補」
を、実際より厳しく（=誤ってng判定）扱っている可能性がある**、つまり
生成品質そのものに悪影響を及ぼしている可能性が高い。

**依存関係：** #2（七句去物実装）はこの解決が前提。それ以外に依存する課題はない。

**見積もり：小〜中。** `candidate`/`history`構築箇所2ファイルに`word:`
（`:plant_type`も、植物のときのみ）を追加するだけの変更自体は小さいが、
D-33-1承認ゲート対象（RengasController変更）であり、また挙動が実際に
変わる可能性が高いため、適用後は本番相当の100句run観測で影響を
定量比較する検証が望ましい（中規模の検証コストがかかる）。

**【対応済み・2026-07-23】D-59-1として本番適用（`c6ac7d5`）。**
`RengasController#create`・`#build_verse_history`・
`script/observe_production_hyakuin.rb`の3箇所で`candidate`/`history`
ハッシュに`:word`（`BuiDictionary#detect_word`新設によるMeCab辞書完全一致語）・
`:text`（全文）・`:plant_type`を追加。`ichiza_violations`の全文照合は
`:text`優先・`:word`フォールバックに変更し、`all_violations`呼び出しに
欠落していた`bui_dict:`引数も補った。ゲートチェック88 pass / 0 failを確認
済み。**ただしこれは配線の実装完了を意味するのみで、「体用区別」「植物異種
間句去」「一座一句物検出」が実際の生成品質にどう影響するかは未検証。**
次はPhase 3（100句規模の本番相当run）で修正前後の挙動を定量比較する
（末尾「Phase 3：本番検証（進行中）」参照）。

### #3　シード辞書の秋偏重・除外後プールの雑優勢

**該当箇所：** `app/services/renga_generator.rb#build_seed_pool`・
`#compute_season_tag`・`SEASON_WORDS`定数、`app/data/bui_dictionary.yml`。

**【重要な訂正】其の五十七・五十八の対応の効果について：** 今回の調査で、
**シードの季節タグ付け（`compute_season_tag`）は`SEASON_WORDS`定数
（renga_generator.rb内のハードコードされた語リスト）のみを参照しており、
`BuiDictionary.yml`の`season:`フィールドは一切参照していない**ことを
確認した。すなわち、其の五十八でBuiDictionary.ymlに追加した
「いとゆふ: season: 春」「帰雁: season: 春」は、**シードプールの季節分布

現状`script/analyze_kigo_membership.rb`（オフライン分析専用スクリプト）
からのみ参照されている（生成パイプラインからは未参照の「半分デッドな
データ」）。#3を実際に前進させるには、`SEASON_WORDS[:spring]`配列に
直接「いとゆふ」「かへるかり」等の文字列を追加する必要がある。

**影響範囲：** 生成される連歌全体の季節バランスに直接影響する、システムの
中核的な品質課題。season_label修正（D-55-1）だけでは秋偏重が解消しな
かったことは其の五十六で確認済み。

**依存関係：** 他の課題との明確な依存はない。#1・#2とは独立に着手可能。

**見積もり：中。** SEASON_WORDS拡充自体は数行の変更で小さいが、効果測定
には100句run規模の観測が必要（過去複数セッションで同様の観測を実施済み、
同程度の工数がかかる見込み）。

### #5　`run_observe_and_summarize.sh`のJST/UTC不整合

**該当箇所：** `script/run_observe_and_summarize.sh:34`（`RUN_DATE=$(date
+%Y%m%d)`、シェルのローカルタイムゾーン依存）と、
`script/observe_production_hyakuin.rb:51`（`Time.zone.now.strftime(...)`、
Rails側）。

**確認した根本原因：** `config/application.rb`の`config.time_zone`は
コメントアウトされたまま（未設定）で、Railsは**デフォルトのUTC**で
動作する。一方シェルの`date`コマンドはMac miniのシステムタイムゾーン
（JST）を使う。両者はJST 00:00〜08:59（UTC前日15:00〜23:59）の間だけ
食い違い、シェル側が予測するjsonlログのファイル名とRuby側が実際に
生成するファイル名が一致しなくなる。

**影響範囲：** サマリー集計スクリプトが「ログが見つかりません」と誤表示
するだけで、observe_production_hyakuin.rb本体の実行・データ保存自体には
影響しない（実害は表示上の誤検知のみ）。

**依存関係：** なし。

**見積もり：小。** `run_observe_and_summarize.sh`の`date`呼び出しを
`date -u`に統一するだけで解消する（1行変更、D-33-1対象外の新規スクリプト
のみの変更）。対応コスト自体は最小だが、発生頻度が低いため急いで直す
動機は薄い。

## Phase 3：本番検証（進行中）

**目的：** D-59-1（#1、ichiza/kuzari配線バグ修正）を本番適用した後の
生成品質への影響を、100句規模の本番相当runで定量観測する。修正前
（sono58以前）のrunログとの比較により、以下を確認する。

- 一座一句物が実際に検出されるようになったか（違反ログの出現有無）。
- 体用（taiyo）区別・植物異種間句去（D-16）が働くことで、以前より
  緩く（正しく）判定される候補が増えたか。逆に、意図せず生成の
  attempt_cap到達やエラー率が悪化していないか。
- #3（シード秋偏重）等、他の既知課題との交絡がないか（同時に走らせて
  いる修正がD-59-1のみであることを確認した上でrunする）。

**実行方法についての注意：** 100句規模のrunは`script/run_observe_and_
summarize.sh <RUN_TAG> [TOTAL_VERSES]`（内部で`bundle exec rails
runner script/observe_production_hyakuin.rb`をtmux経由で長時間実行）
を使う想定だが、過去のセッション（其の三十二）でClaude Code
のBashツール経由で`tmux new-session`すると原因不明にセッションが
消失することがあるため、**このrunはユーザーが直接ターミナルから
起動すること**（Claude Code経由では起動しない）。実行後のログ・
jsonl解析・サマリー集計はClaude Code側で対応可能。

**未着手（次回セッションの最優先タスク）：**
1. RUN_TAGを決めて100句run実行（例：`sono59_run1`）をユーザーが起動。
2. 完走確認後、jsonlログを其の五十六以前と同じ手法で解析し、
   一座一句物・体用区別・植物異種区分の発火有無と、季節分布・
   attempt_cap到達率の変化をD-59-1適用前後で比較する。
3. 結果を`docs/observation_analysis_其の五十九.md`（新規）にまとめ、
   本ファイルの該当課題（#1・#2・#3）のステータスを更新する。
