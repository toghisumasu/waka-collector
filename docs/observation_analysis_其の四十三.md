## 4. 重点確認: D-41-1（build_verse_history二重カウント）修正前後の比較

其の四十一で発見・定量化し、其の四十二で修正した「直前1句の二重カウント」
バグについて、**Run6で同じパターンの誤却下が再発しているか、あるいは解消
されているか**を以下の3段階で検証した。検証用コードは分析専用の一時スクリプト
（`RengasController`本体・DBは無変更、読み取り専用）。

### 4-1. 構造的検証：二重カウントは全99句で再発していない

`RengasController#build_verse_history`（修正後）は次の形になっている：

```ruby
history << { bui: ..., season: ..., verse_type: maeku_type } if chain.empty?
```

Run6の全レコード（previous_renga_idを持つverse2〜100の99句）について、
実際の`fetch_verse_chain`の返り値サイズと`build_verse_history`が返す
historyのサイズを比較した。

| 検証項目 | 結果 |
|---|---|
| 検証対象 | 99句（verse2〜100） |
| history.size ≠ chain.size（＝二重カウント再発）の件数 | **0件** |

`chain.empty?`が真になるのはverse1（previous_renga_idがblank）のみであり、
verse2以降は常にchainが非空（previous_renga_id自体を含む）ため、maekuの
再追加は一度も発生しない。**D-41-1のバグパターンはRun6の全句にわたって
構造的に再発していないことを確認した。**

### 4-2. 過剰却下（over）方向：Run5型の誤却下パターンは0件

Run5のverse34（其の四十一§4-2）のように「本来受理されるべき候補が
streak水増しでkukazo_over誤判定される」パターンが残っていないかを、
Run6の式目ng却下43件全件について**修正後history（実際に使われたもの）**と
**修正前history（旧バグを人為的に再現したもの＝chainが非空でも無条件で
maekuを再追加）**の双方で`ShikimokuChecker#all_violations`を再評価し、
判定（ng/ok）が一致するか比較した。

代表例として、forced_zatsuエスカレーションが発生した4句（verse16・34・62・93）
のkukazo判定を個別に確認：

| verse | 候補季節 | 修正後streak | 修正前(旧バグ)streak | 修正後判定 | 修正前判定 |
|---|---|---|---|---|---|
| 16 | 冬 | 3（超過） | 4（超過） | kukazo_over（streak4/max3） | kukazo_over（streak5/max3） |
| 34 | 冬 | 3（超過） | 4（超過） | kukazo_over（streak4/max3） | kukazo_over（streak5/max3） |
| 93 | 秋 | 5（超過） | 6（超過） | kukazo_over（streak6/max5） | kukazo_over（streak7/max5） |

いずれも修正前後で**両方ともng（streak値は+1ずれるが結論は不変）**。
verse16・34・93のforced_zatsuエスカレーションは、旧バグの有無にかかわらず
発生していたはずの正当な却下であり、**Run5のverse34のような「バグがなければ
本来受理されるべきだった」誤却下パターンはRun6では確認されなかった。**

式目ng全43件を修正後/修正前の両方で再評価した結果、**修正後history=ok
かつ修正前history=ngとなる（＝旧バグが誤って却下していたはずの）ケースは
0件**だった。

### 4-3. 新たな確認：under方向・句去方向の「見逃し」が正しく検出されるようになった

其の四十一§4-2は「二重カウントによりstreakが実際より1大きく見えるため、
最低継続数（kukazo_under、春・秋のみ）方向では**本来ngであるべき遷移が
誤って見逃される（false negative）**可能性がある」という構造的リスクを
指摘していたが、Run5データでは該当事例が確認できていなかった。

Run6の式目ng43件全件で修正後/修正前の判定を比較したところ、**10件で
修正後history=ngだが修正前(旧バグ)history=ok**という逆方向のずれが
見つかった。

| verse | attempt | 候補 | 違反種別 | 修正後判定 | 修正前(旧バグ)判定 |
|---|---|---|---|---|---|
| 28 | 1,2 | 「風の音も心に響きけり」他 | kukazo_under（秋streak2/min3） | ng | ok |
| 36 | 1,2 | 「海の波に願いは消えていく」他 | kukazo_under（秋streak2/min3） | ng | ok |
| 41 | 1 | 「霜の降る夜に 雪の歌かなき」 | kukazo_under（秋streak2/min3） | ng | ok |
| 46 | 1,2,3 | 「風の音に木々が舞い上がる」他 | kukazo_under（秋streak2/min3） | ng | ok |
| 62 | 1 | 「雪の下に眠る花も醒めぬ」 | 句去:植物（kuzari） | ng | ok |
| 79 | 1 | 「星のささやく夜の終わりに」 | kukazo_under（秋streak2/min3） | ng | ok |

これらはすべて「実際のstreakは2（秋が2句しか続いていない＝min3未満で
まだ秋を離れてはいけない）にもかかわらず、旧バグ下ではhistoryが1句
水増しされてstreak3と誤認され、`kukazo_under`違反が見逃されて
受理されていたはずの候補」である。verse62 attempt1の句去（kuzari）
ケースも同じ機構（historyの総長`n`が+1されることで`between`計算がずれる）
による見逃しだった。

**これは其の四十一で「今回のRun5データではverdictが変わる事例は確認され
なかったが、構造的なリスクとして記録する」としていた懸念が、Run6で実際に
10件発生することが確認されたことを意味する。** 言い換えると、**D-41-1修正が
なければ、これら10件の式目違反候補（季節転換の早すぎ8件+句去1件、延べ10行）は
本番シーケンスに誤って組み込まれていた可能性がある。** 逆方向（Run5型の
過剰却下）は今回0件だったため、**修正の正味の効果は「誤却下の解消」よりも
「見逃しの是正」の形でRun6には現れている。**

### 4-4. 結論

| 観点 | Run5（修正前ログ） | Run6（修正後ログ） |
|---|---|---|
| 二重カウントの再発 | ―（当時未修正） | **0/99句、構造的に再発なし** |
| 過剰却下（kukazo_over方向の誤却下） | 8/17件（句数ng却下の47%）が誤却下 | **0件**（該当パターンなし） |
| 見逃し（kukazo_under・kuzari方向） | 0件確認（理論上のリスクとして記録のみ） | **10件確認**（is-ng-but-would-have-been-ok） |

D-41-1修正は意図通りに機能しており、**Run5で確認された「二重カウントに
起因する誤却下」パターン（verse30・34・56相当）はRun6では発生していない
（解消された）**。加えて、修正前は理論上のリスクにとどまっていた
「見逃し（本来ngの候補が誤って通ってしまう）」方向の副作用も、Run6のデータで
初めて具体的に確認された。これはバグ修正の「隠れた便益」であり、D-41-1の
価値を過剰却下の解消だけでなく見逃し防止の面からも裏付けるものと言える。
