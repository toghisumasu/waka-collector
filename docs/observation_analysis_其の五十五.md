## 1. 3-1: ng率比較

集計ロジックは`script/run_observe_and_summarize.sh`のjq定義（total_attempts /
total_ng の対象actionリスト）をそのまま踏襲し、`jq`で再計算した。

```
total_attempts = create + retry + exhausted + forced_zatsu + forced_zatsu_mora_ng + forced_zatsu_create
total_ng       = retry + exhausted + forced_zatsu + forced_zatsu_mora_ng
```

| 指標 | 其の四十三 Run6（D-50-1配線前） | sono51_run1（D-50-1配線後、95句中断） | 差分 |
|---|---|---|---|
| 総試行数 | 159 | 161 | +2 |
| 総ng数 | 62 | 69 | +7 |
| ng率 | 39.0% | **42.9%** | **+3.9pt** |

参考: 其の四十五Run1〜4のベースラインレンジ（31.7%〜40.2%、n=4run）に対しても
sono51_run1の42.9%は範囲外（`docs/architecture_decisions.md`其の五十節で既出の
判定と一致）。

### 差の統計的評価

其の四十五〜四十九で確立した「ベースラインレンジ内か外か」に加えて、二標本比率検定
（正規近似）で補助的に評価した。

```
p1 (Run6)      = 62/159 = 0.390
p2 (sono51_run1) = 69/161 = 0.429
プールした比率 p = 131/320 = 0.409
SE = sqrt(p(1-p)(1/159+1/161)) = 0.0550
z  = (0.429 - 0.390) / 0.0550 = 0.70
```

z=0.70は有意水準5%の目安（|z|≥1.96）を大きく下回り、**この程度の差（+3.9pt）は
n=159/161規模のサンプルでは自然変動の範囲内で、統計的に有意とは言えない**。
一方でベースラインレンジ（31.7%〜40.2%）という運用上の目安には収まっていない
ため、「効果なしと断定できる」わけでもない。**両者は矛盾しない**：レンジ判定は
Run1〜4という限られたn=4回分の観測レンジに対する位置関係を見ているだけであり、
z検定はRun6単体との一対一比較でしかない。いずれの見方でも「明確に悪化した」
とまでは言えないが、「改善した」と言える材料もない、というのが妥当な評価。

なお、sono51_run1は前提0-2の通り外部要因で中断したrunであり、n=161はrun2
（n=155、ng率36.1%、レンジ内）と比べてサンプル自体の性質が異なる（中断が
ng率に系統的な影響を与えた可能性を完全には排除できない）点も付言する。

## 2. 3-2: must_switch分岐の発火確認

### 結論：ログからは判定不能（0件ではなく、計装が存在しない）

`log/observation_sono39_sono51_run1_20260719.jsonl`・`log/observation_stderr_sono51_run1_20260720.log`
のいずれにも`must_switch`・`must_continue`・`season`に関する記述は**1件も存在しない**
（`grep -i`で0件）。

理由は単純で、`must_switch`/`must_continue`をログ出力する`SeasonHintLogger`
（D-52-1）は**其の五十二で新設された機能であり、其の五十一実施のsono51_run1
（2026-07-19〜20）の時点ではまだ実装されていなかった**。つまり「15句サンプルでは
発生しなかった」（其の五十のT5）という状況と異なり、**sono51_run1は「発火が
0件だった」のではなく「発火の有無自体を観測する手段が当時存在しなかった」**。
この違いは依頼書の「発火が0件だった場合」という想定とは別枠として明記しておく。

### 実データ（DB）で確認できる季節遷移

`must_switch`ログはないが、DB上の各句の季節をverse_no順に並べると、以下の
季節遷移が確認できる（`Renga`にseason列はなく、`RengasController#season_from_text`
相当のロジックで句本文から算出。読み取り専用、DB変更なし）。

|---|---|
| 2-3, 5-9, 11-13 | 秋 |
| 14-16 | 春 |

`exhausted`アクションは4件（verse 4, 14, 41, 71）で発生。うちverse4は

### 4-3. 其の二十六の「同一文言固着」パターンとの一致確認

其の二十六（`docs/handover_20260702_其の二十六.md`）で確認された`duplicate_verse`
固着は「5回のリトライ全てで一字一句同一の句を出力し続ける」という強い固着
パターンだった。sono51_run1のログには**この強い固着パターンの直接証拠はない**：

- verse14・41・71の「生成失敗」はいずれも`text:""`（空応答）であり、同一文言の
  繰り返しではない。空応答は`RengaGenerator`内部の25回ループ（5シード×5試行）が
  全滅して`nil`のまま返ってきたことを示すのみで、内部で何が起きていたか
  （固着かモーラ不一致か等）はこのログの粒度では判別できない。
- ただし弱い兆候として、同一verse_no内で同一textが連続する箇所が2件見つかった
  （verse3のattempt2/3「星の下で夢は広がるまま」、verse59のattempt2/3
  「風の音に 時の刻み響く」）。いずれも次の試行で異なる文言に切り替わり
  最終的にcreateへ到達しており、其の二十六のような「5回全滅」までは
  至っていない。

**参考（100in仮説との照合）：** D-51-1（其の五十一、sono51_run2で実施、
本runの2週間前後の別run）の内部診断ログでは、`RengaGenerator`内部却下948件中
category=model（モーラ不一致・echo等）が95%、category=prompt（固着・
history_repeat等、duplicate_verse系はこちらに分類）が5%という内訳だった。
これはsono51_run1本体の診断ではなくsono51_run2で後日追加された別計装の結果
だが、同時期の同一コードパスの参考値として見ると、**waka-collectorの現行
モデル（qwen3:8b）では固着系がモデル出力問題全体の少数派（5%程度）**という
示唆がある。100in（Gemini API）側で観測されている「特定語彙圏への固着から
却下を繰り返しても抜け出せない」現象と同型の弱いパターン（4-3前段の2件）は
sono51_run1にも存在するが、**waka-collector側では固着が「全滅の直接原因」
となる其の二十六当時ほど支配的ではなく、むしろ空応答（generate_fail、
原因未特定）とshikimoku_ng（句数:秋）が全滅要因の主体**という点は、100in側の
仮説（却下理由の事後フィードバック欠如）をそのまま輸入する前に確認しておく
べき差異として記録しておく。

## 5. 3-5: JST/UTC mismatch関連ログの確認（副次的）

sono51_run1のログファイル名自体に、其の五十一で指摘された不整合の**直接証拠**
が残っていた。

| ファイル | 日付部分 | 生成元 |
|---|---|---|
| `observation_sono39_sono51_run1_20260719.jsonl` | **20260719** | `script/observe_production_hyakuin.rb`内`Time.zone.now.strftime("%Y%m%d")` |
| `observation_stderr_sono51_run1_20260720.log` | **20260720** | ラッパー起動コマンドの`date +%Y%m%d`（macOSローカル時刻） |

`config/application.rb`は`config.time_zone`が未設定（コメントアウトのまま）
のため、Railsの`Time.zone`はデフォルトの**UTC**。一方シェルの`date`コマンドは
本機のローカルタイムゾーン（**JST**、`date +%Z`で確認済み）を返す。この2つの
時刻源の食い違いにより、**同一runなのにjsonlログとstderrログのファイル名の
日付が1日ズレている**。

このズレが起きるのはUTCの日付がまだ7/19のうちにJSTでは7/20に入っている時間帯、
すなわち**JST 00:00〜08:59台にrunを開始した場合**であり、依頼書が想定していた
条件と一致する。sono51_run1はまさにこの時間帯に開始・進行していたことになる
（其の五十一の調査でも「クラッシュ時刻09:17:04（JST）」と記録されており、
JST早朝に実行されていたことと整合する）。

なお、`run_observe_and_summarize.sh`側の「ログが見つかりません」というエラー
メッセージが実際に出力されたかどうかは確認できなかった。sono51_run1用の
summaryファイル（`log/observation_sono51_run1_summary.txt`）自体が存在せず
（run2用の`observation_sono51_run2_summary.txt`はある）、これは其の五十一が
記録した「サマリー生成前に外部プロセスごとkillされた」という中断のタイミング
と整合する（サマリー書き出しブロックまで到達していない）。したがって
「エラーメッセージの発生」自体は今回未確認だが、**その根本原因（Time.zone.now=UTC
とシェルdate=JSTの不一致）が実データのファイル名として実証された**という点は
新規の確認事項として記録する。修正は本セッションのスコープ外。

## 6. まとめ：B-4効果について

**判断：「効果ありと言えるか／言えないか」は判断保留（明確な悪化の証拠もない）。**

- ng率は39.0%→42.9%（+3.9pt）で、ベースラインレンジ（31.7%〜40.2%）の外に
  出ているが、二標本比率検定ではz=0.70で統計的に有意な差ではない。
- sono51_run1はn=95・外部要因で中断という固有の事情を持つ1サンプルであり、
  これ単体でB-4（season_hint）の効果を判定する根拠としては弱い。
- 同じD-50-1導入後のsono51_run2（n=155、ng率36.1%）はレンジ内に収まっており、
  「D-50-1導入後の2runのうち1本はレンジ内、1本はレンジ外」という状況は
  其の五十一時点の判断（人間承認済み、運用継続）から変わっていない。
- 「句数:秋」への直接効果（season_hintの本来の狙い）についても、shikimoku_ng
  26件中23件（88%）が「句数:秋」であり、sono51_run1単体を見る限り改善の
  兆候はない。

## 7. 次回セッション（其の五十六以降）への提案

### 7-1. 事後フィードバック機構（100in知見由来）の要否

本分析（§4-3）では、waka-collectorの現行モデルでは強い固着（duplicate_verse、
5回全滅パターン）は主要因ではなく、空応答（generate_fail、原因未特定）と
句数:秋（式目由来）が全滅の主体だった。**100in側の「却下理由が次回プロンプトに
フィードバックされない」という仮説をそのまま輸入するより先に、waka-collector
側の「生成失敗（空応答）」がなぜ起きるか自体の切り分け（D-51-1診断ログの
再導入・拡張、またはgenerate_fail専用の追加計装）が優先度として高いと考える**。
事後フィードバック機構は、固着が支配的要因であることが確認できてから着手する
方が投資対効果が高い。

### 7-2. A系統バックログとの優先順位

`run_observe_and_summarize.sh`のJST/UTC mismatch（§5）は根本原因が今回
実データで裏付けられたため、次点の修正候補として優先度を上げてよいと考える
（`Time.zone.now`をやめて`Time.current`＋明示的なJST変換、またはシェル側の
`date`を`TZ=UTC date`に揃える、のいずれか）。observe_production_hyakuin.rb
rescue範囲見直し等の他のA系統バックログとの相対優先順位は、本分析の範囲では
判断材料が不足しており、別途の依頼書での検討を推奨する。
