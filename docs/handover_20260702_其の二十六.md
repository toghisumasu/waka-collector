## 実施内容の詳細

### 1. 根本原因（duplicate_verse固着）

其の二十五で `describe` のnilクラッシュは修正済みだったが、`:generation_failed`（5回のリトライが全て失敗し `best_candidate` が nil のまま確定するケース）自体がなぜ起きるかは未解明だった。今回 `generation_attempts.log`（試行1回ごとの記録）を新設し、直近5連続dryrun（22:41〜23:36）で発生した `generation_failed` 4件全てを実データで確認した。

**確認した事実（4件とも同型）:**

| 句番目 | history_size | 5attemptの候補文言 | 5attemptのreason |
|:-:|:-:|:--|:--|
| 98 | 97 | 「鶴の声が霞に紛れて響く」（bui申告は毎回微妙に揺れる） | duplicate_verse ×5 |
| 96 | 95 | 「鶴の舞う霞に春の夢は続く」 | duplicate_verse ×5 |
| 68 | 67 | 「鶴の舞う霞の向こうに春の夢かなう」 | duplicate_verse ×5 |
| 92 | 91 | 「鶴の声に混じる春の霞の色」 | duplicate_verse ×5 |

4件とも共通する構造は以下の3点：

1. **duplicate_verse固着** — 5回のリトライ全てで、メンタムさんが**一字一句同一の句**を出力し続け、`history.any? { |v| v[:word] == candidate[:word] }` による重複チェックに毎回弾かれた。
2. **フィードバック無視** — 重複却下時のフィードバックは「全く異なる句を作れ（同じ句は使用不可）」という明示的な指示だが、次のattemptでも寸分違わず同じ文字列を返しており、メンタムさんがこの指示を全く反映していない。
3. **best_candidateがnilのまま生成失敗になる実装** — `dryrun_hyakuin.rb` の重複句リジェクト分岐（`script/dryrun_hyakuin.rb` 重複句チェック節）は `next` するのみで `best_candidate` を一切更新しない。そのため5回とも重複判定で弾かれると `best_candidate` は最後まで nil のままとなり、`{type: :generation_failed}` のプレースホルダ違反が生成される（其の二十五で解明した「nilのままdescribeに渡ってクラッシュしていた」処理の、クラッシュしない側の正常系ルート）。

**つまり `generation_failed` の直接原因は「式目判定側の不備」ではなく、「メンタムさんが同一文言を繰り返し出力し、かつ却下フィードバックを反映しない」というC層のサンプリング不安定性そのものである。**

### 2. bui自己申告に関する訂正

`generation_failure_analysis.md` の作成過程で、同一candidate文言に対して `bui` 判定がattemptごとに揺れる現象（例: 「光物・水辺」→「水辺・降物」）を検出した。当初はこれをB層（`BuiDictionary`/`detect_bui` の形態素解析）側の非決定性ではないかと疑い、`script/test_bui_human.rb` で `BuiDictionary#primary_bui` を直接叩いて確認したが、12語すべて期待通り決定論的に一致した（12 OK / 0 FAIL）。

改めて `script/dryrun_hyakuin.rb` の `parse_candidate` を確認したところ、**このスクリプトの `bui` は `BuiDictionary` を一切参照しておらず、メンタムさんが出力するJSONの `"bui"` フィールドをそのまま採用している**ことを確認した。

**訂正：bui のブレは B層（`BuiDictionary` 参照）の問題ではなく、C層（LLM）が同一の句本文に対して毎回異なる部立を自己申告しているサンプリング不安定性である。** 句去チェックはこの自己申告bui にそのまま依存しているため、句本文自体は妥当でもbui申告がぶれるだけで `kuzari_violation` として却下され続ける構造的リスクがあることも判明した（`log/generation_failure_analysis.md` に70件中39件の該当ケースを記録済み）。

### 3. 集計スクリプトのbui配列順序比較バグ修正

`analyze_generation_failures.rb` の「同一candidateの複数attempt繰り返し検出」において、当初 `Array(e.dig(:candidate, :bui))` を**ソートせずに** `uniq` していたため、`["降物", "水辺"]` と `["水辺", "降物"]` を配列の順序差だけで別物と誤検出し、実際には同一のbui集合であるケースを「bui判定が揺れている」と過大にカウントしていた。

`.sort` を `uniq` の前に挟む（`Array(e.dig(:candidate, :bui)).sort`）よう修正し、要素集合として比較するよう改めた。現行の `script/analyze_generation_failures.rb` はこの修正を反映済み。

### 4. 5回連続dryrunの最終結果（22:41〜23:36）

5回とも完走。`generation_failed` は合計4件（1・2・3回目に集中、4・5回目は0件）。**4件全てにおいて直前句（history最後尾）は式目違反なし（OK）の状態**であり、フィードバックの蓄積による劣化ではなく、メンタムさんの単発的なサンプリング不安定性（同一文言の固着）が直接原因であることが裏付けられた。
