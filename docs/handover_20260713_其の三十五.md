## 実施内容の詳細

### Phase 0：棚卸しで判明した齟齬

依頼書のチェックリストのうち2点は、想定と異なる内容だった：

- **OllamaClient**：`.generate`/`.chat`/`.chat_with_tools`はすでに揃っており、`think`も両メソッドともJSON APIパラメータとして渡されている（raw text prependではない）。「未対応」だったのはOllamaClient自体ではなく、**RengaGeneratorが`.chat`を一切呼んでいなかったこと**
- **duplicate_verse雑句指示**：未反映は事前予想通りだったが、その実装がhistory全体参照を要することが判明し、Phase 3として単独実施できないと判明

RengaChecker軽量化（コミット`0dd2c28`、2026-06-10）とbui自己申告の非移植方針（其の三十三 D-33-1）は、いずれも既存状態が正しく、追加作業は不要と確認した。

### Phase 2：Socratic三段階対話の移植

`app/services/renga_generator.rb`の`generate_tsugeku`内の25回リトライループ（5シード×5attempt）に、`ENDING_BY_MORA`定数と`mora_error_streak`／`past_mora_error_words`／`last_mora_count`のトラッキングを追加した。streakは**generate_tsugeku呼び出し全体（シード再抽選をまたいで）で維持**し、閾値に達すると次attemptで`build_full_prompt`ではなく`socratic_mora_messages`が組み立てる3ターン擬似対話を`OllamaClient.chat`へ渡す構成にした。

ハーネス（dryrun_hyakuin.rb）との違いとして、本番は候補をJSON形式（bui/season/mora自己申告）ではなくプレーンテキスト1行として受け取る設計のため、Socratic対話の最終ターンも「JSON出力せよ」ではなく「一行だけ出力してください」という本番の既存流儀に合わせて調整した。

副産物として、書き換えた同じ行に存在した`is_rep = (ku == seed[:yomi])`の重複代入（其の十八でのコピペミスの名残）を除去した。

### 検証方法の判断

依頼書のゲートコマンド`ruby script/dryrun_hyakuin.rb`は、**dryrun_hyakuin.rbがRengaGeneratorを一切requireしない**独立ハーネスであるため、Phase 2の変更を検証する手段として機能しないことが判明した（其の三十三 D-33-1で確認済みの「別アーキテクチャ」という位置づけが、検証手順の設計にも影響することが明らかになった）。

代わりに以下の方法で検証した：

1. `mora_feedback_message`・`socratic_mora_messages`（private）を`rails runner`から`.send`で直接呼び出し、字余り・字足らず（ENDING_BY_MORA該当あり／なし）の3パターンで出力文言を確認
2. `OllamaClient.generate`/`.chat`を`rails runner`内でシングルトンメソッド差し替えし、意図的にmora不一致を継続させて、streak閾値到達後に正しく`.chat`へ切り替わり、3ターン目で受理可能な候補を返すと正常終了することを確認（呼び出し順序: `[:generate, :generate, :chat, :chat, :chat]`）
3. 実際のOllama（qwen3:8b）を使った`generate_tsugeku`の複数回呼び出し（chouku/tanku各1回）で、通常経路がクラッシュせず自然な付句を返すことを確認
4. `bundle exec rspec`実行時に5件の失敗があったが、変更前（`git stash`で確認）でも同じ5件が失敗しており、本変更と無関係（DB設定起因）と確認済み
