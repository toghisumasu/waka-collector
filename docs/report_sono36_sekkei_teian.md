### 決定的な追加知見：`build_verse_history`の`chain.size < 9`が観測結果と噛み合わない

dryrun観測で検知した24件のDUPを「相手句との距離（何句前か）」で再集計すると：

| 距離（何句前） | 該当件数 |
|---|---|
| 9句以内 | **11件（46%）** |
| 10句以上（最大82句前） | **13件（54%）** |

**案Aの土台である`build_verse_history`は`chain.size < 9`に固定**されており、そのまま流用すると観測された逆戻りの半分以上（54%）を最初から捕捉できない。これは「式目の直近連鎖チェック」用に設計された深さであり、「一巻を通じた語彙の逆戻り検知」には深さの設計思想自体が異なる。案Aを採る場合、この定数も変更する必要があり、その時点で「既存スケルトンの流用」というA案の利点（差分の小ささ）はほぼ消える。

## 2. Levenshtein距離計算のコスト試算

実測ログの句（14〜18文字程度）を模した文字列で、履歴99句全走査を1000回試行してベンチマークした。

```
history=99句、1000回試行の合計: 1.9682秒
1句あたり(履歴99句全走査): 1968.1 マイクロ秒（約2ミリ秒）
```

**結論：無視できる。** 本番の1attempt（Ollama呼び出し）は数秒〜数十秒かかっており（`Rails.logger.info "[RengaGenerator] attempt: #{Time.now - gen_start}s"`で既に計測実績あり）、履歴全件との逐次比較コスト（~2ms）はその1000分の1以下である。1句あたり最大25回のattempt（5シード×5回）がすべて全履歴スキャンを行っても合計50ms程度で、深さを99に制限せず全履歴を持つ設計にしてもコスト面の懸念はない。

## 3. 対処ロジック統合の設計案

### 3-1. 「完全一致固着」と「類似検知」を別メカニズムにしない

現状の整理で分かった通り、本番には「完全一致固着」の対処自体が存在しない。したがって新規実装にあたり、両者を**最初から1つのLevenshtein判定に統合**するのが自然である。

- 距離0 → 「完全一致」
- 距離 ≤ `max(文字数×0.3, 3)` → 「類似・一語違い」
- どちらも同じ判定関数・同じ履歴走査で一度に処理する（ハーネスで検証済みのロジックをそのまま移植）

### 3-2. トリガー条件の合流点

既存の2段階パターン（mora_error_streakで実証済み）をそのまま踏襲する：

| 段階 | 条件 | 対処 |
|---|---|---|
| 軽量feedback | 履歴との一致を初検知（streak内1回目） | 既存の`feedback`機構（`build_full_prompt`の`feedback_line`）に「既出につき別の語で」を流す。`is_sticky`と同じ扱い方 |
| Socratic雑句レスキュー | 同一verse_no内で連続2回以上、履歴一致（完全/類似いずれでも可） | 其の三十五で実装済みのchatベース3段階対話（`renga_generator.rb:419-437`）を**流用**し、内容だけ「雑（無季）へ」から「既出表現を避けよ」に差し替えた新バリアントを追加する |

完全一致と類似で**閾値だけ変え、機構は共有**すれば実装コストは小さく抑えられる（新規コードは主に判定ロジック＋streakカウンタ＋feedback文言の3点、いずれもmora_error_streakと相似形）。

### 3-3. 検知タイミング

`generate_tsugeku`内、`is_echo`/`is_rep`/`is_sticky`判定（`renga_generator.rb:166-186`）と同じ位置に追加する。この位置は既にmora検査後・violation検査前であり、動作順序を変えずに済む。

## 4. 推奨案（決定ではなく推奨）

**案C（段階的接続）を推奨する。** 理由：

1. コード差分は案Bと同一で最小・低リスク
2. `build_verse_history`という「6/26に作られて以来未接続のまま放置されていた」実例（其の三十六 Phase 0で発覚）を踏まえると、"後で統合する"という口約束だけでは同じ運命を辿りかねない。案Cは統合先延ばしを**明示的な計画として残す**ことでこれを予防する
3. 案Aの土台（`chain.size < 9`）が観測データと数値的に噛み合わないことが判明した以上、いま流用するメリットがない
4. フェーズ8（式目/bui分類の本実装）の設計がまだ固まっていない段階で、性急に一本化すると、フェーズ8側の設計を重複検知の都合に引きずられて歪める恐れがある

ただし、Bおよびcontrollerに「chain歩行ロジックが2本になる」ことの居心地の悪さは事実であり、他のトレードオフを優先するなら案A/Bも合理的な選択である。

## 5. 未確定・要人間判断の論点

1. **履歴の深さ**：無制限（一巻全体、最大99句）か、それとも上限を設けるか。コスト面では無制限が可能（§2）だが、`previous_renga_id`を毎回1句ずつ辿るDBクエリ回数（N+1）は別途考慮が必要（Levenshteinの計算コストとは別の話）
2. **閾値`max(文字数×0.3, 3)`の妥当性**：ハーネスの1回のdryrunで検証しただけの値。本番投入後にfalse positive（誤検知）/false negative（見逃し）の傾向を見て調整する運用にするか、先に複数dryrunで閾値を振って比較するか
3. **完全一致と類似のfeedback文言をどこまで変えるか**：同じSocratic機構を使うとして、LLMに提示するメッセージ強度（「既出なので別の語で」vs「似すぎているので別の語で」）を分けるかどうか
4. **N+1クエリ対策**（案A/B/C共通）：`previous_renga_id`を辿る現行の`build_verse_history`ループ自体、1句ごとに`Renga.find_by`を呼ぶ実装（`rengas_controller.rb:79-83`）。履歴を深くする場合、このN+1が式目チェックと重複検知の両方で発生することになり、パフォーマンス上の追加検討事項になり得る（今回のLevenshteinベンチマークはRuby内計算のみでDBアクセスを含まない点に注意）
